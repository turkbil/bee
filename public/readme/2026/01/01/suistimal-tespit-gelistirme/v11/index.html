Reddedilen Yaklaşımlar (Ne Öğrendik)

Spotify Modeli (Real-time Blocking)

Kullanıcı: "Hayır istemiyorum. Tespiti kendimiz nasıl yaparız."

Neden Reddedildi:

  • • Real-time engelleme istenmiyor
  • • Sadece tespit ve raporlama yeterli
  • • Mevcut sistemi geliştirmek hedef

Coğrafi Analiz (IP → Şehir/Ülke)

Kullanıcı: "Coğrafi analizi falan siktir et. Eldeki verilerle ne yapılabilir ona odaklan."

Neden Reddedildi:

  • • Ekstra API/servis gerektiriyor
  • • Mevcut veriler yeterli
  • • Basit tutmak hedef

5 Kural + Puan Sistemi (5+ Cihaz Eşiği)

Kullanıcı: "5+ cihaz, 5+ IP eşiği çok toleranslı. B2B'de 1 üye = 1 cihaz olmalı. 5 kural puanı da saçma."

Neden Reddedildi:

  • • 5 cihaz toleransı B2B için çok fazla
  • • 1 abonelik = 1 cihaz olmalı
  • • Puan sistemi gereksiz karmaşıklık
  • • VAR/YOK mantığı daha basit

Bot/Robot Tespiti (Yüksek Hacim + Skip Yok)

Kullanıcı: "Otomasyon bot bizi kullanmaz. Zaten tüm müşteriler paralı. Restoran 15 saat, 7/24 dinler. Bizi ilgilendirmez."

Neden Reddedildi:

  • • B2B model = paralı müşteriler
  • • Bot kullanma motivasyonu yok
  • • Restoran 15 saat, 7/24 dinler = NORMAL
  • • Yüksek hacim + skip yok = B2B'de normal

Terminoloji Düzeltmesi: "Robot" → "Split Stream"

Kullanıcı: "Hop, robot dinleme değil bu. Bu da 2 aynı PC'den özel sistemle 2 ayrı hoparlöre yayın demek."

Düzeltme:

  • • "Robot Listening" yanlış terminoloji
  • • Doğrusu: "Split Stream" (1 PC → 2 hoparlör)
  • • Tespit: Aynı fingerprint'ten OVERLAP

Sonuç:

Universal çözümler incelendi, bizim B2B modele göre sadeleştirildi. Basit 3 pattern yeterli.

Temel Kural

1 Abonelik = 1 Cihaz = 1 Aktif Yayın

2 hoparlöre ayrı yayın = 2 abonelik gerektirir

Kullanıcı: "1 üye = 1 cihaz = 1 session = 1 limit = 1 tarayıcı"

B2B İş Modeli

✅ Müşteri Profili

  • • Tüm müşteriler paralı (ücretli abonelik)
  • • Restoranlar, kafeler, işletmeler
  • • Arka plan müziği için kullanım
  • • Bot/otomasyon kullanma motivasyonu yok

⚡ Normal Kullanım

  • • Sabah aç, gece kapat = 15 saat aralıksız
  • • Hatta 7/24 çalabilir
  • • Skip yok, şarkılar sonuna kadar çalar
  • • Yüksek hacim = NORMAL

Kontrol ETMİYORUZ (Normal Kullanım)

Yüksek Hacim

15 saat = normal

Tutarlı Süreler

Şarkı sonuna kadar = normal

Skip Yok

Arka plan müziği = normal

Bot Tespiti

Paralı müşteri = bot yok

Bunlar B2C platformlarda (Spotify vb.) kontrol edilir. B2B'de gereksiz.

Tespit Edilecek Suistimal Türleri (3 Pattern)

1. Ping-Pong Pattern

Herhangi bir veride A → B → A döngüsü = Hesap paylaşımı

Kullanıcı: "Ping-pong'un her türlüsü suistimaldir. IP, tarayıcı, device vs. her türlü elimizdeki verinin ping-pong'u"

Kontrol Edilecek Veriler:

  • • ip_address
  • • browser (Chrome, Safari, Firefox...)
  • • platform (Windows, macOS, Linux...)
  • • device_type (Desktop, Mobile, Tablet)

Suistimal Örnekleri:

Chrome → Safari → Chrome ❌

Windows → macOS → Windows ❌

IP-A → IP-B → IP-A ❌

Desktop → Mobile → Desktop ❌

Not: IP dinamik olabilir. Tek başına IP değişimi yeterli değil, döngü (A→B→A) olmalı.

Kullanıcı: "IP adresini de değiştirebilir. Static IP kullanmaz. En önemli husus ping-pong'da."

2. Eş Zamanlı Farklı Kaynak

Aynı anda 2 farklı fingerprint = 2 farklı lokasyon

Kullanıcı: "X aralıkta sürekli aynı anda 2 farklı IP = hata"

Fingerprint = IP + Browser + Platform

Şarkı A:

10:00-10:05 | IP: 92.44.x | Chrome | Windows

Şarkı B (aynı anda):

10:02-10:07 | IP: 88.230.x | Safari | macOS

→ 2 farklı lokasyondan eş zamanlı dinleme = SUİSTİMAL

3. Çoklu Yayın (Split Stream)

Aynı PC'den özel yazılımla 2 ayrı hoparlöre yayın

Kullanıcı: "Bu da 2 aynı PC'den özel sistemle 2 ayrı hoparlöre yayın demek"

Nasıl Çalışıyor?

1 PC

Özel Yazılım

(2 browser tab)

Hoparlör 1

Hoparlör 2

Tespit Kriteri: SADECE OVERLAP

Aynı fingerprint'ten (IP + Browser + Platform) aynı anda 2 şarkı çalıyorsa:

Şarkı A: 10:00 - 10:03 | fingerprint X

Şarkı B: 10:01 - 10:04 | fingerprint X (aynı!)

→ Aynı PC'den 2 paralel stream = SUİSTİMAL

Tüm Senaryolar (Kullanıcıdan Gelen)

SUİSTİMAL Senaryoları

1

X aralıkta sürekli aynı anda 2 farklı IP

→ 2 farklı lokasyondan eş zamanlı kullanım

Sonuç: SUİSTİMAL

2

X aralıkta bir IP durur, diğeri başlar SIRAYLA (döngü)

→ IP-A → IP-B → IP-A → IP-B... (ping-pong)

Sonuç: SUİSTİMAL (hesap paylaşımı)

3

Aynı anda 2 farklı device/browser/platform

→ Chrome + Safari aynı anda, Desktop + Mobile aynı anda

Sonuç: SUİSTİMAL

4

Herhangi bir veride ping-pong döngüsü

→ Browser, platform, device, IP... herhangi birinde A→B→A

Sonuç: SUİSTİMAL (2 kişi sırayla kullanıyor)

5

Aynı fingerprint'ten aynı anda 2 şarkı (overlap)

→ 1 PC, 2 browser tab, 2 farklı hoparlöre yayın

Sonuç: SUİSTİMAL (split stream)

NORMAL Senaryolar

1

Biri durur, diğeri başlar KALICI (döngü yok)

→ IP-A → IP-B (ve IP-B'de kalır, geri dönmez)

Sonuç: NORMAL (lokasyon değiştirdi, taşındı)

Kullanıcı: "Biri durup diğeri başlarsa sorun yok"

2

15 saat aralıksız dinleme

→ Restoran sabah açar, gece kapatır

Sonuç: NORMAL (B2B kullanım)

3

7/24 kesintisiz dinleme

→ Bazı işletmeler hiç kapatmaz

Sonuç: NORMAL (B2B kullanım)

4

Hiç skip yok, şarkılar sonuna kadar çalıyor

→ Arka plan müziği, kimse müdahale etmiyor

Sonuç: NORMAL (restoran kullanımı)

5

Tutarlı şarkı süreleri (hep sonuna kadar)

→ 3:24, 2:58, 4:12... hep tam süre

Sonuç: NORMAL (playlist otomatik çalıyor)

6

Dinamik IP değişimi (döngü olmadan)

→ ISP zaman zaman IP değiştirir

Sonuç: NORMAL (A→B→C→D... tek yönlü)

Kritik Notlar

IP Hakkında

  • • IP dinamik olabilir
  • • Static IP kullanmak zorunda değil
  • • Tek başına IP değişimi = suistimal değil
  • • Önemli olan DÖNGÜ (ping-pong)

Döngü vs Kalıcı Geçiş

  • • A → B → A = SUİSTİMAL (döngü)
  • • A → B (kalıcı) = NORMAL (taşındı)
  • • Fark: Geri dönüyor mu?
  • • Döngü sayısı önemli (2+ = şüpheli)

Bot Tespiti Neden Yok?

  • • Tüm müşteriler paralı
  • • B2B model = otomasyon motivasyonu yok
  • • Restoran 15 saat dinler = bot gibi görünür ama değil
  • • "Yüksek hacim + skip yok" = B2B'de normal

Fingerprint

  • • Fingerprint = IP + Browser + Platform
  • • Aynı fingerprint = aynı cihaz/lokasyon
  • • Farklı fingerprint = farklı cihaz/lokasyon
  • • Overlap kontrolü fingerprint bazlı

Final Özet

Pattern Tespit Yöntemi Anlam Sonuç
Ping-Pong A → B → A döngüsü (IP, browser, platform, device) 2 kişi sırayla kullanıyor SUİSTİMAL
Eş Zamanlı Farklı Aynı anda 2 farklı fingerprint 2 farklı lokasyon SUİSTİMAL
Çoklu Yayın Aynı fingerprint + OVERLAP (aynı anda 2 şarkı) 1 PC → 2 hoparlör SUİSTİMAL
Kalıcı Geçiş A → B (geri dönmez) Lokasyon değiştirdi NORMAL
Uzun Dinleme 15 saat, 7/24, yüksek hacim Restoran kullanımı NORMAL
Skip Yok Şarkılar sonuna kadar, tutarlı süreler Arka plan müziği NORMAL
