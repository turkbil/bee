Problem Tanımı

Basit Anlatım

Tüm müşteriler aynı sunucuda. Sunucumuz küçük. Büyük müşteriler (gazeteler, hastaneler) kendi sunucularını kullanmak istiyor. Hem küçüklere ortak hizmet, hem büyüklere kendi sunucularında çalışma imkanı vermemiz lazım. Ayrıca ana sunucu çökerse diğerleri etkilenmemeli.

Teknik Detaylar

Mevcut Sorunlar:

  • • Tek sunucu = Tek hata noktası (SPOF)
  • • Sunucu kapasitesi sınırlı
  • • Büyük müşteriler için yeterli kaynak yok
  • • Ana sunucu çökerse herkes etkileniyor

Hedefler:

  • • Farklı müşteri tiplerine farklı çözüm
  • • Self-hosted müşteri desteği
  • • Ayrı License Server (SPOF çözümü)
  • • Lisans koruması (bypass engellemeli)

Lisans Koruma Sistemi

KRİTİK - v9

Basit Anlatım

Birisi "lisans kontrolünü kaldırayım bedava kullanayım" diyemeyecek. Çünkü sistem sadece "lisans geçerli mi?" diye sormayacak, aynı zamanda çalışması için gereken bazı şeyleri lisans sunucusundan alacak. Lisans kodu silinirse veya değiştirilirse sistem çalışmayı durduracak.

Problem: Lisans Bypass

Kötü niyetli müşteri şunu yapabilir:

// Middleware/CheckLicense.php
public function handle($request, $next) {
    // return $this->verifyLicense(); // ← KALDIRIR!
    return $next($request);           // ← DİREKT GEÇİŞ!
}

Sonuç: Lisans kontrolü atlanır, yazılım bedava kullanılır!

Koruma Katmanları

1

Çoklu Kontrol Noktası

Lisans kontrolü tek yerde değil, birden fazla kritik noktada yapılır. Birini kaldırsan diğeri yakalar.

  • • Middleware (her request)
  • • ServiceProvider boot (uygulama başlangıcı)
  • • Scheduler (her dakika)
  • • Kritik controller'lar (admin, API)
2

Kod Bütünlüğü Kontrolü (Checksum)

Kritik dosyaların hash'i lisans sunucusunda saklanır. Dosya değiştirilirse sistem çalışmaz. 

// Lisans sunucusu kontrol eder:
{
  "app/Http/Middleware/CheckLicense.php": "a1b2c3...",
  "app/Services/LicenseService.php": "d4e5f6...",
  "app/Providers/LicenseServiceProvider.php": "g7h8i9..."
}
3

Kritik Veri Sunucudan Gelir

Bazı özellikler için gereken config/key değerleri lisans sunucusundan gelir. Lisans yoksa bu değerler alınamaz, özellik çalışmaz.

  • • Şifreleme anahtarları (encryption keys)
  • • Premium özellik flag'leri
  • • API rate limit değerleri
4

Obfuscation (Kod Karmaşıklaştırma)

Kritik PHP dosyaları ionCube veya benzeri araçlarla şifrelenir. Kod okunmaz, değiştirilemez.

Opsiyonel - Ekstra güvenlik katmanı. Müşteri sunucusunda ionCube loader gerektirir.

5

Anomali Tespiti

Şüpheli davranışlar tespit edilir: Lisans kontrolü hiç gelmezse, heartbeat durmuşsa uyarı.

  • • 24 saat verify gelmezse → Uyarı
  • • Heartbeat tamamen durmuşsa → İnceleme
  • • Farklı IP'den istek → Log

Korumalı Lisans Doğrulama Akışı

// 1. Müşteri sunucu → Lisans sunucuya istek
POST license.tuufi.com/api/license/verify
{
    "license_key": "TUUFI-XXX",
    "domain": "musteri.com",
    "file_hashes": {
        "CheckLicense.php": "a1b2c3...",
        "LicenseService.php": "d4e5f6..."
    }
}

// 2. Lisans sunucu kontrol eder:
✓ Lisans geçerli mi?
✓ Domain eşleşiyor mu?
✓ Dosya hash'leri doğru mu? ← YENİ!

// 3. Yanıt
{
    "valid": true,
    "expires": "2027-01-01",
    "runtime_key": "xyz789...",  ← Kritik! Her doğrulamada yeni
    "features": ["blog", "shop", "music"]
}

// 4. runtime_key olmadan bazı özellikler çalışmaz!

Koruma Sonucu

  • ❌ Lisans kodu silinirse → Hash uyuşmaz → Çalışmaz
  • ❌ Lisans kodu değiştirilirse → Hash uyuşmaz → Çalışmaz
  • ❌ Verify atlanırsa → runtime_key alınamaz → Kritik özellikler çalışmaz
  • ❌ Offline çalışırsa → 7 gün sonra runtime_key expire → Çalışmaz
