Problem Tanımı

Basit Anlatım (Herkes İçin)

Şu anda tüm müşteriler aynı sunucuda çalışıyor. Sunucumuz küçük. Bazı müşteriler (gazeteler, hastaneler) zaten kendi sunucularına büyük paralar ödüyor. Bu müşteriler bizim yazılımı kendi sunucularında kullanmak istiyor. Biz de hem küçük müşterilere ortak sunucudan hizmet verip, hem de büyük müşterilerin kendi sunucularında çalışmasını sağlayacak bir sistem kurmak istiyoruz.

Teknik Detaylar

Mevcut Sorunlar:

  • • Tek sunucu = Tek hata noktası (SPOF)
  • • Sunucu kapasitesi sınırlı
  • • Büyük müşteriler için yeterli kaynak yok
  • • Trafik artınca herkes etkileniyor

Hedefler:

  • • Farklı müşteri tiplerine farklı çözüm
  • • Self-hosted müşteri desteği
  • • Merkezi lisans kontrolü
  • • Otomatik güncelleme mekanizması

Müşteri Tipleri

Basit Anlatım

İki ana müşteri tipimiz var: Küçükler bizim sunucumuzu kullanıyor (ixtif.com gibi), Büyükler kendi sunucularını kullanıyor (gazeteler, hastaneler gibi). Büyükler zaten sunucuya 500-1000$/ay veriyor, bize sadece yazılım lisansı ödüyorlar.

TİP A: SHARED

Ortak Sunucu

Bizim sunucumuzda çalışır
Mevcut Stancl/Tenancy yapısı
Bakım/güncelleme bizden
Düşük giriş maliyeti

Örnekler: ixtif.com, küçük işletmeler, startup'lar

TİP B: SELF-HOSTED

Kendi Sunucusu

Müşteri sunucusunda çalışır
Lisans anahtarı ile doğrulama
Kendi trafiği, kendi storage'ı
Otomatik git pull güncelleme

Örnekler: Ulusal gazeteler, hastaneler, Muzibu, büyük şirketler

Gerçek Hayat Senaryoları

Ulusal Gazete

Zaten ayda 500-1000$ sunucu parası ödüyor. Yüzbinlerce haber, milyonlarca trafik. Kendi sunucusunda çalışacak, bize lisans ücreti ödeyecek.

Hastane

KVKK/HIPAA uyumu gerekiyor. Veriler kendi datacenter'ında kalmalı. İzole sistem, dış dünyadan erişim yok.

Muzibu

Şu an shared'da başlıyor. Büyüyünce 1 origin + 2 edge sunucu yapısına geçecek. Kademeli büyüme modeli.

İxtif (Küçük Site)

Bizim sunucuda kalıyor. Düşük trafik, düşük maliyet. Shared tenant olarak devam.

Lisans Sistemi

Basit Anlatım

Her müşteriye benzersiz bir anahtar veriyoruz. Müşteri sunucusu günde bir kez "bu anahtar geçerli mi?" diye bize soruyor. Geçerliyse çalışmaya devam ediyor. 7 gün boyunca bize ulaşamasa bile çalışmaya devam eder (grace period).

1

Lisans Anahtarı

Her müşteriye benzersiz anahtar

TUUFI-GZTE-2026-A1B2-C3D4
2

Doğrulama API

Günde 1 kez lisans kontrolü

POST /api/license/verify
3

Grace Period

Offline çalışma süresi

7 gün (cache'lenmiş lisans)

Lisans Doğrulama Akışı

Müşteri Sunucu

POST /verify

tuufi.com kontrol

OK + Versiyon

Otomatik Güncelleme Sistemi

Basit Anlatım

Sen kodu güncelleyip GitHub'a yüklediğinde, tüm müşteri sunucuları otomatik güncelleniyor. Müşteri hiçbir şey yapmak zorunda değil. Sistem her dakika kontrol edip yeni versiyon varsa git pull yapıyor.

update.sh (Müşteri Sunucusunda Çalışır) 

#!/bin/bash

# 1. Maintenance mode aç
php artisan down

# 2. Git pull
git pull origin main

# 3. Composer
composer install --no-dev

# 4. Migration
php artisan migrate --force

# 5. Cache
php artisan config:cache
php artisan route:cache

# 6. Maintenance mode kapat
php artisan up

# 7. Başarı bildirimi
curl -s "https://tuufi.com/api/license/heartbeat?key=$LICENSE_KEY"

Shared Müşteriler

Bizim sunucuda, otomatik güncellenir.

# Biz yaparız
git pull && php artisan migrate

Self-Hosted Müşteriler

Kendi sunucularında, cron ile otomatik.

# Cron her dakika kontrol eder
* * * * * /var/www/site/check-update.sh

Webhook ile Anında Güncelleme

Basit Anlatım

Sen "git push" dediğin an GitHub bize haber veriyor. Biz de versiyon numarasını güncelliyoruz. Müşteri sunucuları her dakika "yeni versiyon var mı?" diye kontrol ediyor. Varsa hemen güncelleniyor. Push'tan 1 dakika sonra 100 müşteri güncel!

Tam Akış

1. git push 2. GitHub webhook 3. tuufi.com alır 4. version güncelle 5. müşteri poll (1dk) 6. git pull ✓ Güncel!

Toplam süre: Push'tan ~1 dakika sonra tüm sunucular güncel

Neden Direkt Webhook Değil de Polling?

Webhook Push Sorunları:

  • ❌ Müşteri firewall'ı açmalı
  • ❌ Endpoint güvenliği sağlanmalı
  • ❌ Sunucu offline olursa webhook kaybolur

Polling Avantajları:

  • ✅ Firewall sorunu yok (outbound)
  • ✅ Sunucu online olunca güncelleme alır
  • ✅ Daha güvenli

API Endpoints

POST /api/license/verify

Lisans doğrulama - Müşteri sunucu günlük çağırır

Request:

{
  "license_key": "TUUFI-...",
  "domain": "musteri.com",
  "version": "2.4.8"
}

Response:

{
  "valid": true,
  "expires": "2027-01-01",
  "features": ["blog", "shop"]
}
GET /api/version

Son versiyon kontrolü - Polling için

Response:

{
  "current_version": "2.5.1",
  "released_at": "2026-01-01 15:30:00",
  "changelog_url": "https://..."
}
POST /api/license/heartbeat

Güncelleme başarı bildirimi

Request:

{
  "license_key": "TUUFI-...",
  "version": "2.5.1",
  "status": "updated"
}
POST /api/webhook/github-deploy

GitHub webhook - Push bildirimini alır

İşlem:

  1. 1. GitHub signature doğrula (HMAC-SHA256)
  2. 2. Branch kontrol (sadece main)
  3. 3. system_info.current_version güncelle

Müşteri Kurulum Süreci

Basit Anlatım

Müşteri lisans satın alır, biz anahtar veririz. Müşteri IT ekibi sunucuya kodu kurar, .env'ye anahtarı ekler, cron ayarlar. Bitti! Artık otomatik güncelleme alacak.

1

Lisans Satın Alma

Müşteri ödeme yapar, biz licenses tablosuna kayıt ekleriz, mail ile anahtar göndeririz.

2

Deploy Key Oluşturma

Müşteri SSH key oluşturur, public key'i bize gönderir:

ssh-keygen -t ed25519 -C "deploy@musteri.com"
3

Git Clone

Biz deploy key'i repo'ya ekleriz (read-only), müşteri clone yapar:

git clone git@github.com:tuufi/platform.git /var/www/site
4

.env Ayarları

TUUFI_LICENSE_KEY=TUUFI-GZTE-2026-XXXX
DB_DATABASE=musteri_db
DB_USERNAME=musteri_user
5

Cron Ayarı

# Her dakika güncelleme kontrolü
* * * * * /var/www/site/check-update.sh

# Laravel scheduler
* * * * * php /var/www/site/artisan schedule:run

Tamam!

Sistem çalışır durumda. Artık otomatik güncelleme alacak.

Fiyatlandırma Modeli

Müşteri Tipi Sunucu Lisans Modeli Destek
SHARED Bizim sunucu Aylık düşük ücret Email
SELF-HOSTED Müşteri sunucusu Yıllık yüksek ücret Email + Öncelikli
ENTERPRISE Müşteri datacenter Yıllık + SLA 7/24 + Telefon

Önemli Not

Self-hosted müşteri zaten sunucuya 500-1000$/ay veriyor. Senden sadece yazılım lisansı alıyor. Bu yüzden lisans ücreti shared'dan yüksek olabilir - müşteri için yine de ucuz çünkü kendi altyapısını kullanıyor.
